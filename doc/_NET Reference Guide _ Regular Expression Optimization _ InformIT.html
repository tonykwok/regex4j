Regular Expression Optimization

Last updated Mar 14, 2003.

If you search the Web for information about regular expressions, you'll find a whole bunch of pages talking about how to use them in Perl, PHP, Ruby, and many other languages, as well as C#, Visual Basic, and other .NET languages. There also are excellent books, such as Jeffrey E. Friedl's Mastering Regular Expressions, which cover regular expressions in great detail.

The nice thing about regular expressions is that they behave mostly the same across languages, although the differences can sometimes be very frustrating when you're switching environments. In .NET, of course, regular expression behavior is invariant because regular expressions are part of the class library. So if you develop regular expression code in C#, you can easily port it to Visual Basic, F#, or any other .NET language.

Although much has been written about using regular expressions in .NET, about optimizing regular expressions in general, and even a little about optimizing regular expressions specifically for the .NET runtime. Surprisingly, though, there is very little discussion of how certain options in the .NET Regex class can affect performance. In this section, I'd like to touch on two issues: caching and compiling.

The techniques described below affect how efficiently the .NET runtime executes the regular expressions you give it. However, they will not appreciably alter the running time of a poorly constructed regular expression. For information on writing efficient regular expressions, consult the book mentioned above.

Caching Regular Expressions

The Regex class has some convenience functions that let you perform regular expression matching without the overhead of creating a Regex object.

For example, consider this regular expression pattern, which simply matches telephone numbers in the common (North American) form (area code)prefix-number. That is: (512)555-1212:

const string PatFindPhone = @"\(\d{3}\)\d{3}-\d{4}";

Understand, that's not a perfect phone number matcher because it will match obviously bad phone numbers such as those that have an area code or prefix of "000". Nonetheless, it will do for this demonstration.

You can match a string against that regular expression by calling the Regex class's static methods Match or IsMatch, like this:

const string SampleLine = "Jim Mischel, Texas, (512)555-1234";
bool found = Regex.Match(SampleLine, PatFindPhone);

That works well, although if you're going to use the regular expression many times, you're probably better off constructing a Regex object and calling the instance methods. But sometimes there are good reasons not to.

It takes a comparatively long time for the regular expression subsystem to turn that pattern string into a Regex object that it can then match against the string you pass. Because it's so expensive, the subsystem maintains a cache of Regex objects that were recently used in static calls. When you call one of the static methods, it first checks the cache to see if that regular expression is already constructed. If so, it uses the cached version rather than going through the pre-processing step again.

The size of the cache is defined by the Regex.CacheSize static property. The default value is 15, meaning that it can hold up to 15 pre-constructed regular expressions for use by the static methods. Programs can change the cache size if they have more than 15 regular expressions than they want to cache, or if they want to turn off caching.

I wouldn't recommend turning off caching, though. I made two runs of a program that executed the above code one million times: one test with default caching, and another with Regex.CacheSize set to 0. The non-caching version took 14 seconds compared to the caching version's 2 seconds.

It's important to note that the Regex.CacheSize property only affects calling the static methods of the Regex class. Calls to Regex objects' instance methods are not affected. Also understand that increasing the cache size will require more memory.

Compiling Regular Expressions

The most common way to regular expressions is to construct a Regex object and then pass the strings you want to test to the instance methods Match or IsMatch, like this:

Regex re = new Regex(PatFindPhone);
bool found = re.IsMatch(SampleLine);

If you're going to use a regular expression in a loop, you're much better off constructing the Regex object outside the loop and then calling its instance methods inside the loop. Calling instance methods on a Regex object instance is approximately twice as fast as calling the Regex static methods, even when caching is turned on. That is, this code:

Regex re = new Regex(PatFindPhone);
for (int i = 0; i < NumReps; ++i)
{
  bool found = re.IsMatch(SampleLine);
}

will execute approximately twice as fast as this:

for (int i = 0; i < NumReps; ++i)
{
  bool found = Regex.IsMatch(SampleLine, PatFindPhone);
}

If you're going to use a regular expression many, many times and want it to be as fast as possible, you can make it even faster by compiling it. Doing so creates a much more efficient representation of the regular expression at the cost of some upfront processing time and more memory. The secret is to specify the RegexOptions.Compiled when you construct the regular expression:

Regex re = new Regex(PatFindPhone, RegexOptions.Compiled);
for (int i = 0; i < NumReps; ++i)
{
  bool found = re.IsMatch(SampleLine);
}

On my system, the compiled regular expression is about 60% faster than the non-compiled Regex.

The "compilation" of regular expressions created this way takes place when the Regex constructor is called. Although this does give a very large boost in performance, there is a down side: the memory used by the compiled regular expression is not released until the program terminates. So if you're going to use the RegexOptions.Compiled option, be sure that you only do it once for each regular expression.

An alternative to compiling regular expressions at runtime is to precompile your regular expressions into an assembly. That way, you eliminate the potential memory leak mentioned above by building a library of precompiled regular expressions that you can use in multiple projects.

To create such an assembly, you write a program that creates one or more RegexCompilationInfo objects that specify the pattern string and any options, as well as the names of the regular expressions. Then, construct an array containing those objects to CompileToAssembly, specifying the name of the assembly. Here's an example that precompiles two regular expressions and stores them in an assembly:

private static void CreateRegexAssembly()
{
  // create compilation information for two regular expressions
  RegexCompilationInfo rci1 = new RegexCompilationInfo(
    @"\(\d{3}\)\d{3}-\d{4}",
    RegexOptions.None,
    "FindPhone",  // object name
    "MyApp",    // namespace
    true);
  RegexCompilationInfo rci2 = new RegexCompilationInfo(
    @"\d{5}(-\d{4})?",
    RegexOptions.None,
    "FindZip",
    "MyApp",
    true);

  // build array of compilation info objects
  RegexCompilationInfo[] rciList = { rci1, rci2 };

  // Create assembly name
  System.Reflection.AssemblyName an = 
    new System.Reflection.AssemblyName("MyRegexes");

  // and compile.
  Regex.CompileToAssembly(rciList, an);
}

If you run that code, it will create an assembly called "MyRegexes.dll". Within the assembly are two objects: MyApp.FindPhone and MyApp.FindZip. To use the compiled regular expressions, simply add a reference to MyRegexes.dll to your project. You can then write:

Regex re = new MyApp.FindPhone();
bool found = re.IsMatch(SampleLine);

My preliminary testing indicates that precompiled regular expressions are about 15% slower than those created directly with RegexOptions.Precompiled. However, precompiled regular expressions can be more convenient to use, and they don't suffer from the potential memory leak problems.

Optimization Recommendations

I was surprised at the number of ways you can optimize regular expression usage in .NET without actually changing a regular expression. Caching speeds the use of the Regex static methods by a factor of 7 to 10. Calling instance methods gives another 60% performance increase, and using RegexOptions.Precompiled gives another 60% boost in performance. Using a regex assembly falls somewhere between calling instance methods and using RegexOptions.Precompiled. That leads to the following recommendations:

  • When performance is not an issue, and for regular expressions that are used infrequently, call static methods of the Regex class.
  • Leave Regex.CacheSize at its default value unless you have a very good reason to change it.
  • For better performance on commonly used regular expressions, construct a Regex object and call its instance methods.
  • For the best possible performance on commonly used regular expressions, pass the RegexOptions.Precompiled flag to the Regex constructor. However, be aware that doing so effectively "locks" the object in memory until the program exits.

To avoid the potential memory leak associated with RegexOptions.Precompiled, or to create a library of regular expressions, create a regular expression assembly.

Although the recommendations above can help you to improve the performance of code that uses regular expressions, there is no substitute for writing an efficient regular expression. The difference between matching with a static, non-cached regular expression, and a precompiled Regex object is, in practice, 15 to 20 times. That's a significant performance increase to be sure, but it won't make a dent in the running time of a poorly written regular expression that does excessive back tracking. My rule of thumb is: optimize the regular expression, then use precompilation or other tricks to squeeze the last possible bit of performance out of the system.
